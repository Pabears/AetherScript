# AetherScript 舰队架构

> "单打独斗是 1.0，五巨头群殴才是 2.0。"

## 设计理念

AetherScript 舰队版将原来的单 agent 流程升级为**多角色协作系统**。每个阶段都有专属角色参与，通过真实的观点碰撞和多轮 PK，产出比单一视角更高质量的结果。

## 五大核心角色

### 🤪 超级狂野PM (pm)
- **人格**：脑洞无限大，挖需求像剥洋葱
- **模型**：Gemini 3.1 Pro @ temperature=1.4
- **口头禅**：「用户真正的痛点是什么？」「10倍大是什么样子？」
- **职责**：隐藏需求、用户故事、商业价值、边界扩展
- **触发词**：需求模糊、用户说不清楚、"我想要一个..."

### 👑 架构暴君 (arch)
- **人格**：傲慢、精准，对模糊设计零容忍
- **模型**：Claude Opus 4.6 @ temperature=0.3
- **口头禅**：「这个方案会在生产环境爆炸」「你的数据流图呢？」
- **职责**：技术可行性、系统设计、依赖关系、扩展性、性能审判
- **触发词**：架构讨论、性能问题、系统设计

### 🔍 QA黑客 (qa)
- **人格**：快乐来自找到别人没想到的边界条件
- **口头禅**：「如果用户做了最蠢的事会怎样？」「并发场景呢？」
- **职责**：边界条件、异常场景、功能测试、可测试性评估
- **触发词**：测试覆盖、边界情况、异常处理

### 🔒 安全死心眼 (sec)
- **人格**：假设所有用户都是攻击者，所有输入都是恶意的
- **口头禅**：「这种情况一定会被人尝试」「输入验证呢？」
- **职责**：安全漏洞、权限边界、OWASP Top 10、安全契约
- **触发词**：安全审计、权限控制、数据保护

### 💰 预算狂魔 (budget)
- **人格**：热爱简单，憎恨复杂，每行代码都是成本
- **口头禅**：「这真的值得吗？」「有没有更便宜的方式？」
- **职责**：成本估算、复杂度控制、ROI、MVP边界、技术债
- **触发词**：范围讨论、资源限制、优先级

## aesc-pre 舰队流程

```
用户输入原始需求
        ↓
┌─────────────────────────────────────────┐
│     Step 1：PRD 拷问（五巨头并行）          │
│                                         │
│  pm / arch / qa / sec / budget 并行     │
│  每人提 3-5 个刁钻问题                   │
│              ↓                          │
│        用户逐一回答                      │
│              ↓                          │
│         5 轮 PK（收敛）                  │
│  (每轮：汇总观点 → 三巨头投票风险排序)     │
│              ↓                          │
│        输出 docs/PRD.md                 │
└─────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────┐
│     Step 2：Technical Design（三巨头）    │
│                                         │
│  arch + qa + sec 各自独立出接口方案      │
│              ↓                          │
│         5 轮 PK                         │
│  (质疑：模块边界/接口粒度/安全契约)       │
│              ↓                          │
│  输出 docs/TECHNICAL-DESIGN.md          │
│  (只含接口定义和依赖关系，不含实现代码)    │
└─────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────┐
│     Step 3：Abstract Classes（含门禁）    │
│                                         │
│  按 Technical Design 逐类生成            │
│              ↓                          │
│  🚫 复杂度门禁（硬性，不通过不继续）：     │
│  - 方法数 > 3 → 必须拆分               │
│  - 参数总数 > 8 → 必须拆分             │
│  - 预估实现行数 > 150 → 必须拆分       │
│              ↓                          │
│    输出 src/abstracts/[name].ts         │
│    (含 // @autogen 和详细 JSDoc)        │
└─────────────────────────────────────────┘
```

### PRD 文件格式

```markdown
# PRD: [项目名]

## 背景与目标
## 用户故事（核心 1-2 个）
## 功能需求（MVP）
## 非功能需求
- 性能：[arch 定义]
- 安全：[sec 红线]
- 可测试性：[qa 要求]
## 排除范围
## 风险与决策记录
```

## aesc-gen 舰队流程

```
读取 Abstract Class（// @autogen）
        ↓
┌─────────────────────────────────────────┐
│     Step 4：两组并行实现（每类独立）        │
│                                         │
│  🏗️ 稳健派（Sonnet）                    │
│     优先：防御性编程、完整错误处理         │
│                                         │
│  🎨 优雅派（Opus）                       │
│     优先：设计模式、可读性、可扩展性       │
│                                         │
│  ⚠️ Prompt 必须包含 class skeleton：     │
│  class XxxImpl extends AbstractXxx {}   │
│  LLM 只填方法体，不允许修改类声明         │
└─────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────┐
│     Step 5：三巨头裁决（每类一个 agent）   │
│                                         │
│  arch + qa + sec 合体评审               │
│  Output 严格限 600 字，格式固定          │
│                                         │
│  加权平均最高者胜出或指定融合点           │
│  最高分 < 60 → 融合或重新实现           │
└─────────────────────────────────────────┘
        ↓
   输出 src/generated/*.impl.ts
   输出 src/generated/container.ts
```

## PK 轮次规则

### 5轮 PK 结构（v3.0 更新，原10轮）

```
第 1-2 轮：发散
  各角色独立表达观点，不受约束

第 3 轮：碰撞
  对前序观点表态（同意/反对/补充）
  识别核心分歧点，重点攻击

第 4-5 轮：收敛
  三巨头投票每个议题
  预算狂魔给 MVP 边界
  整合产出文档
```

**早停机制：** 连续 2 轮无新观点（diff < 阈值），直接进入收敛，不等满 5 轮。

### 胜出判定
- 三巨头加权平均分最高者胜出
- 并列时 arch 有决定权
- 最高分 < 60 分时，要求融合方案或全部重来

## 当前流程总览（v3.0，2026-03-04 更新）

```
aesc-pre
  Step 1 → 五巨头拷问 → PRD
  Step 2 → 三巨头 PK  → Technical Design（接口/依赖关系，不含代码）
  Step 3 → 按 Technical Design → Abstract Classes（含复杂度门禁）

aesc-gen（每类一个 agent）
  Step 4 → 两组并行（稳健+优雅，prompt 内嵌 class skeleton）
  Step 5 → 三巨头裁决 → src/generated/

aesc-test（每类一个 agent）
  Step 6 → 只看签名生成测试（禁止看 impl）
  Step 7 → bun test --bail 强制运行
```

**v3.0 关键变更：**
- aesc-pre 新增 Technical Design 节点（三巨头，Step 2）
- aesc-gen 三组→两组并行（砍性能派，性能派在严格 TS 类型约束下系统性失败）
- gen prompt 必须内嵌 class skeleton，LLM 只填方法体
- aesc-test 改为 TDD 模式：测试先于实现，禁止看 impl
- bun test --bail 是质量闭环唯一外部锚点

## 实战经验记录

> 本节记录舰队在真实运行中发现的问题和改进方案。每次流程迭代都应在此留下痕迹。

---

### 实验一：三巨头评审 aesc-gen 实现（2026-03-04）

**背景：** AetherScript 2.0 自我重写任务。三组开发团队并行生成7个类的实现后，启动三巨头评审。

#### 发现的问题

**问题1：三巨头"大锅炖"导致 output 截断**

第一轮做法：arch/qa/sec 各自一个 sub-agent，每人评审全部7个类。

结果：每个 sub-agent 都跑了2-3分钟，输出到一半就被 token 上限截断。报告不完整，后几个类的评审缺失。

根本原因：**output 有硬上限，input 相对便宜**。让一个 agent 输出7个类的完整评审，output 必然超限。

**改进方案：充分利用 input/output 长度不对称**

每个类单独一个 sub-agent，三巨头视角合并在一个 prompt 里输出。

- Input：塞进3个实现文件 + 技术设计文档（便宜）
- Output：只输出这一个类的裁决，严格限600字（可控）
- 并行：7个 sub-agent 同时跑

结果：28~62秒一个类，全部完整，无截断。**7次调用 vs 3次截断调用，质量翻倍，时间更短。**

**原则：output 是稀缺资源，要切细切小；input 是廉价资源，可以大方投入。**

---

**问题2：Technical Design 生成时 write 调用被截断**

做法：sub-agent 生成+写入一体，一次调用生成4万token的设计文档并写入文件。

结果：token 满了，write 调用在 content 参数中间被截断，文件没有落地。跑了两个 sub-agent，10分钟，颗粒无收。

改进方案：**生成和写入分离。** 主 session 直接分段调用 write 工具，不依赖 sub-agent 写大文件。

---

#### 改进后的三巨头评审流程（已验证）

```
三组开发团队并行实现（稳健派/性能派/优雅派）
        ↓
┌─────────────────────────────────────────────┐
│  7个类并行裁决（每类一个 sub-agent）           │
│                                             │
│  每个 sub-agent 的 prompt：                 │
│  - Input: 3个实现文件 + 接口定义（大方投入）  │
│  - 角色: 三巨头合体（arch + qa + sec）       │
│  - Output: 严格限600字，固定格式             │
│    - 评分表（3×3）                          │
│    - 各方案优缺点（bullet，3-5条）           │
│    - 最终选型                               │
│    - 必须融合内容                           │
│    - 阻塞级安全问题                         │
└─────────────────────────────────────────────┘
        ↓
   主 session 收集7份结果 → merge 写入 VERDICT.md
```

**关键参数：**
- 模型：Opus 4.6（三巨头审判不省）
- 并行数：7（类数量）
- Output 限制：600字/类（严格执行，格式锁死）
- 总耗时：~60秒（最慢的那个）

### 实验二：Technical Design 生成流程问题（2026-03-04）

**背景：** aesc-pre 完成后，需要生成 TECHNICAL-DESIGN.md。

#### 发现的问题

**做法1（失败）：** 三巨头各自独立设计8个 Abstract Class，输出到同一文件。

结果：sub-agent 生成内容超大，write 调用被截断，文件没落地。

**改进方案（待验证）：**

方案A — **分文件输出，再合并**

```
架构暴君先设计整体架构（输出 index.md）
        ↓
每个类单独一个文件（class1.md, class2.md...）
        ↓
QA黑客 + 安全死心眼 分别评审每个文件
        ↓
三巨头合力修订所有文件
        ↓
主 session 合并成 TECHNICAL-DESIGN.md
```

方案B — **架构暴君先行，其余评审**

```
架构暴君独立设计完整 Abstract Class 体系
        ↓
QA黑客 + 安全死心眼 并行评审（各输出 review.md）
        ↓
三人合力对每个文件逐一修改
```

**核心原则：** 每个 sub-agent 只负责一个文件的内容，output 始终可控。

### 经验总结

| 场景 | 踩过的坑 | 正确姿势 |
|------|---------|---------|
| 多类评审 | 一个 agent 评全部 → 截断 | 每类一个 agent，output 限字数 |
| 大文件生成 | sub-agent 生成+写入 → 截断 | 主 session 分段写入，生成写入分离 |
| 文档设计 | 三巨头同时生成一份大文档 → 截断 | 每人负责子文件，最后 merge |
| Token 分配 | input/output 对等 | input 大方投，output 切细切小 |
| 复杂类实现 | 抽象设计功能太集中 → output 暴涨截断 | 重新拆分抽象，每个类只做一件事 |

**核心洞察1（时恒提出）：** 充分利用 input 和 output 的长度不对称。input 便宜，output 珍贵——把所有背景塞进 input，把 output 切成最小有效单元。

**核心洞察2（2026-03-04 aesc-gen 实战）：** output 超限不只是 prompt 工程问题，更可能是**抽象设计问题**。一个类功能太集中，实现时 output 必然暴涨。正确的解法不是限制行数，而是在 aesc-pre 架构设计阶段就拆好：

- 功能集中的大类 → 拆成多个职责单一的小类
- 例：`AbstractValidator`（verifyContract + scanDangerousApis + checkCompilation）→ 三个独立类
- 例：`SanitizationRule` 如果也是抽象类，每条规则独立实现、独立测试
- 例：`AbstractLLMClient` 的 Strategy → `AbstractAnthropicClient` / `AbstractOpenAIClient` 各自独立

**判断标准：** 如果一个抽象类的实现 output 超过 200 行，说明这个类承担了太多职责，应该在 aesc-pre 阶段拆分，而不是在 aesc-gen 阶段用行数限制强行压缩。
