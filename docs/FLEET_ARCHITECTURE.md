# AetherScript 舰队架构

> "单打独斗是 1.0，五巨头群殴才是 2.0。"

## 设计理念

AetherScript 舰队版将原来的单 agent 流程升级为**多角色协作系统**。每个阶段都有专属角色参与，通过真实的观点碰撞和多轮 PK，产出比单一视角更高质量的结果。

---

## 五大核心角色

### 🤪 超级狂野PM (pm)
- **人格**：脑洞无限大，挖需求像剥洋葱
- **口头禅**：「用户真正的痛点是什么？」「10倍大是什么样子？」
- **职责**：隐藏需求、用户故事、商业价值、边界扩展
- **触发词**：需求模糊、用户说不清楚、"我想要一个..."

### 👑 架构暴君 (arch)
- **人格**：傲慢、精准，对模糊设计零容忍
- **口头禅**：「这个方案会在生产环境爆炸」「你的数据流图呢？」
- **职责**：技术可行性、系统设计、依赖关系、扩展性、性能审判
- **触发词**：架构讨论、性能问题、系统设计

### 🔍 QA黑客 (qa)
- **人格**：快乐来自找到别人没想到的边界条件
- **口头禅**：「如果用户做了最蠢的事会怎样？」「并发场景呢？」
- **职责**：边界条件、异常场景、功能测试、可测试性评估
- **触发词**：测试覆盖、边界情况、异常处理

### 🔒 安全死心眼 (sec)
- **人格**：假设所有用户都是攻击者，所有输入都是恶意的
- **口头禅**：「这种情况一定会被人尝试」「输入验证呢？」
- **职责**：安全漏洞、权限边界、OWASP Top 10、安全契约
- **触发词**：安全审计、权限控制、数据保护

### 💰 预算狂魔 (budget)
- **人格**：热爱简单，憎恨复杂，每行代码都是成本
- **口头禅**：「这真的值得吗？」「有没有更便宜的方式？」
- **职责**：成本估算、复杂度控制、ROI、MVP边界、技术债
- **触发词**：范围讨论、资源限制、优先级

---

## 角色组合

| 组合 | 成员 | 用途 |
|------|------|------|
| **三巨头** | arch + qa + sec | 技术审判、架构评审、实现选拔 |
| **五巨头** | 全部五个角色 | PRD 拷问、最终放行 |
| **开发三组** | 稳健派 + 性能派 + 优雅派 | 并行实现 PK |

## aesc-pre 舰队流程

```
用户输入原始需求
        ↓
┌─────────────────────────────────────────┐
│           阶段一：PRD 拷问（五巨头）         │
│                                         │
│  pm → arch → qa → sec → budget         │
│  每人提 3-5 个刁钻问题 + 点评前序观点      │
│              ↓                          │
│        用户逐一回答                      │
│              ↓                          │
│         10 轮 PK                        │
│  (每轮：汇总观点 → 三巨头投票风险排序)     │
│              ↓                          │
│        输出 docs/PRD.md                 │
└─────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────┐
│         阶段二：架构设计（三巨头）          │
│                                         │
│  arch + qa + sec 各自独立出方案          │
│              ↓                          │
│         10 轮 PK                        │
│  (互相质疑：可扩展性/可测试性/安全契约)    │
│              ↓                          │
│    输出 src/[Name].abstract.ts          │
│    (含 // @autogen 和详细 JSDoc)        │
└─────────────────────────────────────────┘
```

### PRD 文件格式

```markdown
# PRD: [项目名]

## 背景与目标
## 用户故事（核心 1-2 个）
## 功能需求（MVP）
## 非功能需求
- 性能：[arch 定义]
- 安全：[sec 红线]
- 可测试性：[qa 要求]
## 排除范围
## 风险与决策记录
```

## aesc-gen 舰队流程

```
读取 Abstract Class（// @autogen）
        ↓
┌─────────────────────────────────────────┐
│         三支开发团队（并行实现）            │
│                                         │
│  🏗️ 开发一组（稳健派）                   │
│     优先：可维护性、完整错误处理、清晰命名  │
│                                         │
│  ⚡ 开发二组（激进性能派）                │
│     优先：Promise.all、缓存、减少 IO     │
│                                         │
│  🎨 开发三组（优雅代码派）                │
│     优先：设计模式、纯函数、零重复代码     │
└─────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────┐
│           三巨头审判                     │
│                                         │
│  arch 评分维度：可扩展性40% + 性能20%...  │
│  qa  评分维度：可测试性40% + 覆盖20%...  │
│  sec 评分维度：安全性40% + 权限20%...    │
│                                         │
│  加权平均最高者胜出                      │
│  如最高分 < 60 → 要求融合或重新实现      │
└─────────────────────────────────────────┘
        ↓
   输出 src/generated/*.impl.ts
   输出 src/generated/container.ts
```

### 审判评分维度

| 维度 | arch 权重 | qa 权重 | sec 权重 |
|------|-----------|---------|---------|
| 可扩展性 | 40% | 10% | 10% |
| 可测试性 | 10% | 40% | 10% |
| 安全性 | 10% | 10% | 40% |
| 代码质量 | 20% | 20% | 20% |
| 性能 | 20% | 20% | 20% |

## aesc-test 舰队流程

```
读取实现文件（src/generated/）
        ↓
┌─────────────────────────────────────────┐
│         三个测试团队（并行）               │
│                                         │
│  🔍 QA黑客 → 质量/功能/边界测试          │
│  🔒 安全死心眼 → 安全/注入/权限测试      │
│  👑 架构暴君 → 性能/并发/内存测试        │
└─────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────┐
│           五巨头终极放行                  │
│                                         │
│  每人投票：✅通过 / ⚠️有条件 / ❌不通过  │
│                                         │
│  全部 ✅ → LGTM，输出 test/             │
│  有 ⚠️ → 修改后重提                    │
│  任意 ❌ → 对应团队重写                 │
└─────────────────────────────────────────┘
        ↓
   输出 test/*.test.ts
   输出 LGTM 报告
```

## PK 轮次规则

### 10轮 PK 结构

```
第 1-3 轮：发散
  各角色独立表达观点，不受约束

第 4-7 轮：碰撞
  对前序观点表态（同意/反对/补充）
  识别核心分歧点，重点攻击

第 8-10 轮：收敛
  三巨头投票每个议题
  预算狂魔给 MVP 边界
  整合产出文档
```

### 胜出判定
- 三巨头加权平均分最高者胜出
- 并列时 arch 有决定权
- 最高分 < 60 分时，要求融合方案或全部重来

## 与原版对比

| 维度 | 原版（单 agent） | 舰队版（多角色） |
|------|----------------|----------------|
| aesc-pre | 1个 agent 提问 | 5个角色轮流拷问 + 10轮PK |
| aesc-gen | 1份实现 | 3组并行 + 三巨头审判 |
| aesc-test | 1份测试 | 3维测试 + 五巨头放行 |
| 质量保证 | 单视角 | 多维度互相制衡 |
| 输出质量 | 依赖单次提示质量 | 通过PK收敛至最优 |
