# AetherScript 2.0 PRD

> 版本：1.0 | 日期：2026-03-04 | 状态：五巨头收敛定稿

---

## 1. 项目背景与目标

**一句话定位：** AetherScript 是一个 AI 驱动的 TypeScript 代码生成框架，从抽象类定义自动生成带 DI 注册的实现代码。

**2.0 解决的核心问题：** 1.0 用 5+ 次串行 LLM 调用、零结构化校验、无安全门禁完成了一个 1-2 次调用就能解决的任务。2.0 的目标是**在保留可观测阶段分界的前提下，将 LLM 调用压缩至 ≤2 次，补齐结构化校验与安全门禁，使框架从"能跑的 demo"变成"可信赖的工具"。**

---

## 2. 目标用户

| 用户画像 | 具体描述 | 核心诉求 |
|---|---|---|
| **主力用户** | 中高级 TS 开发者，已有项目使用 DI（inversify/tsyringe 等），团队规模 3-15 人 | 从接口定义到骨架实现的自动化，减少样板代码，保证 DI 注册不出错 |
| **集成用户** | CI/CD 管道、代码生成脚本的维护者 | 可预测的执行时间、确定性输出、明确的失败信号（exit code ≠ 0） |

**明确排除：** 不面向"零 TypeScript 经验的小白"——框架假设用户能写抽象类和 DI 配置。

---

## 3. 核心设计原则

1. **逻辑阶段 ≠ LLM 调用次数。** 保留 scan → generate → validate → write 的逻辑分界，但允许多个逻辑阶段共享同一次 LLM 调用。
2. **结构化输出，不做正则英雄。** LLM 输出必须是 JSON Schema 约束的结构化数据。解析失败 = 阶段失败，不做"尽力提取"。
3. **安全审查必须是 Gate，不是 Comment。** 安全校验失败必须阻断流水线（exit ≠ 0），不是"打印一条警告然后继续"。
4. **生成代码写入前必须通过 AST 校验。** 不能编译 = 不写入。不符合接口契约 = 不写入。
5. **成本与 token 是一等公民。** 每次执行必须报告 token 用量和预估成本。超过成本阈值必须中断并提示。

## 4. 功能需求

### 4.1 保留（骨架不动）

| 模块 | 说明 |
|---|---|
| `aesc_scan` | 扫描 `// @autogen` 标记的抽象类，提取类型上下文。 |
| `aesc_write` / `aesc_test_write` | 写入 `src/generated/` 和 `test/` 的文件操作。保留接口，内部加校验。 |
| 逻辑阶段分界 | scan → generate → validate → write 四阶段日志和事件钩子保留。 |

### 4.2 改造（血肉换）

#### LLM 调用合并（P0）
- **调用 1（主生成）：** 单次调用，输入完整类型上下文，输出 JSON，包含实现代码 + 单元测试 + DI container 注册代码
- **调用 2（安全 + 质量 Gate）：** 独立调用（防止自我确认偏差），输入生成的代码，输出结构化 pass/fail 判定

#### 结构化输出（P0）
使用 JSON Mode / Tool Use 强制结构化输出，解析失败触发 1 次自动重试，重试仍失败则阶段终止。

#### 安全门禁（P0）

| 校验项 | 实现方式 | 失败行为 |
|---|---|---|
| 接口契约校验 | AST 解析，校验是否实现所有方法签名 | 阻断，报告缺失方法 |
| 危险 API 扫描 | AST 扫描 eval、child_process、非白名单 fs 调用 | 阻断，报告违规行 |
| 编译校验 | tsc --noEmit | 阻断，报告编译错误 |
| container.ts 原子写入 | 读取→AST 合并→整体写回（不是追加） | 消除竞态 |

#### 错误恢复（P1）
事务模式：先写临时目录，全部校验通过后原子移动。失败时临时文件保留在 `.aesc-tmp/` 供调试。

### 4.3 新增

| 功能 | 优先级 | 说明 |
|---|---|---|
| Token/成本报告 | P0 | 每次执行结束输出 token 用量、预估成本 |
| 成本熔断 | P1 | 配置 `maxCostPerRun`（默认 $0.50），超出则中断 |
| `--dry-run` 模式 | P1 | 全流程执行但不写文件 |
| 阶段事件钩子 | P2 | `onScanComplete` 等回调，供 CI 集成 |

## 5. 非功能需求

| 类别 | 指标 | 目标 |
|---|---|---|
| 性能 | LLM 调用次数 | ≤ 2（正常），≤ 3（含重试） |
| 性能 | 端到端执行时间（10 个抽象类） | ≤ 30s（1.0 基线 ~90s） |
| 成本 | 单次 token 消耗 | ≤ 50K tokens |
| 成本 | 相对 1.0 成本降低 | ≥ 60% |
| 可靠性 | 结构化输出解析成功率 | ≥ 95%（含 1 次重试） |
| 可靠性 | 编译通过率 | ≥ 90% 首次，≥ 98% 含重试 |
| 可靠性 | 幂等性 | 相同输入连续执行 2 次，输出 diff = 0 |

---

## 6. 明确不做的事（Out of Scope）

| 不做 | 原因 |
|---|---|
| 多模型同源辩论（五巨头内部辩论） | 同源模型多角色是伪多样性，ROI 为负 |
| 运行时代码热加载 | 生成代码是静态文件，不在生成阶段执行 |
| 非 TypeScript 语言支持 | 2.0 聚焦 TS + DI 场景 |
| IDE 插件 | 留给 3.0 |
| 多轮自动修复失败生成 | 成本不可控，重试 1 次后报错退出 |
| 替代现有 DI 框架 | 生成注册代码，不自带容器 |
