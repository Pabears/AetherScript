# AetherScript 2.0 PRD

> 版本：3.0 | 日期：2026-03-04 | 状态：十巨头评审后收敛定稿（工作流重设计）

---

## 1. 项目背景与目标

**一句话定位：** AetherScript 是一个 AI 驱动的 TypeScript 代码生成框架，从抽象类定义自动生成带 DI 注册的实现代码。

**核心问题（持续迭代）：**
- 1.0 → 2.0：串行调用太多、零结构化校验、无安全门禁
- 2.0 → 3.0（本版本）：舰队工作流经过实战暴露了三个根本性问题——
  1. **三组并行是伪多样性**：性能派 7/7 全灭（不继承抽象类），1/3 成本打水漂
  2. **抽象类粒度无强制检查**：功能集中的抽象类导致实现 output 截断，根因在上游 aesc-pre 没有复杂度门禁
  3. **质量闭环没合上**：测试从未真正运行，AI 审 AI 没有外部锚点

**3.0 目标：** 在保留舰队协作价值的前提下，修复工作流架构缺陷——加复杂度门禁、改并行策略、引入 TDD、让测试真正运行。

---

## 2. 目标用户

| 用户画像 | 具体描述 | 核心诉求 |
|---|---|---|
| **主力用户** | 中高级 TS 开发者，已有项目使用 DI（inversify/tsyringe 等），团队规模 3-15 人 | 从接口定义到骨架实现的自动化，减少样板代码，保证 DI 注册不出错 |
| **集成用户** | CI/CD 管道、代码生成脚本的维护者 | 可预测的执行时间、确定性输出、明确的失败信号（exit code ≠ 0） |

**明确排除：** 不面向"零 TypeScript 经验的小白"——框架假设用户能写抽象类和 DI 配置。

---

## 3. 核心设计原则

1. **逻辑阶段 ≠ LLM 调用次数。** 保留 scan → generate → validate → write 的逻辑分界，但允许多个逻辑阶段共享同一次 LLM 调用。
2. **结构化输出，不做正则英雄。** LLM 输出必须是 JSON Schema 约束的结构化数据。解析失败 = 阶段失败，不做"尽力提取"。
3. **安全审查必须是 Gate，不是 Comment。** 安全校验失败必须阻断流水线（exit ≠ 0），不是"打印一条警告然后继续"。
4. **生成代码写入前必须通过 AST 校验。** 不能编译 = 不写入。不符合接口契约 = 不写入。
5. **成本与 token 是一等公民。** 每次执行必须报告 token 用量和预估成本。超过成本阈值必须中断并提示。
6. **抽象类粒度是工作流的前置门禁。** 方法数 > 3 或实现预估 > 150 行的抽象类，必须在 aesc-pre 强制拆分，不得进入 aesc-gen。（新增 v3.0）
7. **gen prompt 必须内嵌 class skeleton。** LLM 只填方法体，class 声明（含 extends/implements）由框架提供，不允许 LLM 修改。防止"不继承抽象类"类错误。（新增 v3.0）
8. **测试必须先于实现、独立于实现。** 测试生成的 input 只有抽象类签名+JSDoc，禁止看到实现代码。生成后必须真正运行（`bun test --bail`），不过则阻断。（新增 v3.0）
9. **质量闭环需要外部锚点。** AI 审 AI 不是质量保证。每个类的实现必须通过编译+运行测试这两个确定性、不可被幻觉欺骗的校验点。（新增 v3.0）

## 4. 功能需求

### 4.1 保留（骨架不动）

| 模块 | 说明 |
|---|---|
| `aesc_scan` | 扫描 `// @autogen` 标记的抽象类，提取类型上下文。 |
| `aesc_write` / `aesc_test_write` | 写入 `src/generated/` 和 `test/` 的文件操作。保留接口，内部加校验。 |
| 逻辑阶段分界 | scan → generate → validate → write 四阶段日志和事件钩子保留。 |

### 4.2 改造（血肉换）

#### aesc-pre：复杂度门禁（P0，新增 v3.0）
aesc-pre 产出抽象类后，**自动运行复杂度检查**，不通过不进 aesc-gen：
- 方法数 > 3 → 强制拆分
- 参数总数 > 8（全部方法合计）→ 强制拆分
- 依赖类型数 > 5 → 建议拆分（warning）
- 预估实现行数 > 150 → 强制拆分

拆分不是"行数限制"，是设计问题：一个类做了太多事，必须在设计阶段解决。

#### aesc-gen：并行策略改造（P0，修改 v3.0）

**旧策略（废弃）：** 三组并行（稳健/性能/优雅）+ 三巨头裁决

**新策略：**
- **gen prompt 必须内嵌 class skeleton**：框架提供 `class XxxImpl extends AbstractXxx implements Xxx { ... }` 骨架，LLM 只填方法体。不允许修改类声明、继承关系、方法签名。
- **两组并行**：稳健派（Sonnet，防御优先）+ 优雅派（Opus，可读性优先）。砍掉性能派——在严格 TypeScript 类型约束下，"性能优先"容易导致契约违反。
- **三巨头 review**：两组实现分别由 arch/qa/sec 评审，选优或融合。

#### LLM 调用（P0）
- **调用 1（主生成）：** 单次调用，输入完整类型上下文 + class skeleton，输出 JSON，包含实现代码 + DI container 注册代码
- **调用 2（安全 + 质量 Gate）：** 独立调用，使用与生成不同家族的模型（防同源确认偏差），输出结构化 pass/fail 判定

#### 结构化输出（P0）
使用 JSON Mode / Tool Use 强制结构化输出，解析失败触发 1 次自动重试，重试仍失败则阶段终止。

#### aesc-test：TDD 模式（P0，修改 v3.0）

**旧策略（废弃）：** 实现完成后生成测试，测试 input 包含实现代码

**新策略：**
1. **测试先行**：在 aesc-gen 之前，只看抽象类签名+JSDoc 生成测试用例
2. **测试独立于实现**：禁止测试生成时看到实现代码（防止 AI 把 bug 当 feature 测）
3. **测试必须真正运行**：`bun test --bail` 通过才算完成。不运行的测试 = 不存在的质量保证
4. **测试覆盖门禁**：每个 `.impl.ts` 必须有对应 `.test.ts`，且测试通过，否则阻断流程

#### 安全门禁（P0）

| 校验项 | 实现方式 | 失败行为 |
|---|---|---|
| 接口契约校验 | AST 解析，校验是否实现所有方法签名 | 阻断，报告缺失方法 |
| 危险 API 扫描 | **AST 扫描**（非正则），eval/exec/new Function/动态 import/globalThis | 阻断，报告违规行 |
| 编译校验 | tsc --noEmit | 阻断，报告编译错误 |
| 测试运行 | bun test --bail | 阻断，报告失败用例 |
| container.ts 原子写入 | 读取→AST 合并→整体写回（不是追加） | 消除竞态 |

#### 安全设计修复（P0，新增 v3.0）

实战发现的安全层设计缺陷，必须在抽象层修复：

| 缺陷 | 修复方案 |
|---|---|
| `SANITIZED_SYMBOL` 被 export，可被 import 伪造 | 改为模块私有，只通过 `verify()` 暴露，不 export Symbol 本身 |
| Scanner 无 deny-list，安全组件可被标记 @autogen | Scanner 硬编码拒绝安全组件的 autogen 标记（Validator/Sanitizer/Scanner/FileWriter/Pipeline） |
| 危险 API 检测用正则，可被字符串拼接绕过 | 升级为 AST 级别分析（ts-morph）|
| JSDoc 内容无结构化清洗，是一等公民注入面 | Sanitizer 剥离/转义 JSDoc 中的指令性语言，再送入 LLM |

#### 错误恢复（P1）
事务模式：先写临时目录，全部校验通过后原子移动。失败时临时文件保留在 `.aesc-tmp/` 供调试。

## 6. 明确不做的事（Out of Scope）

| 不做 | 原因 |
|---|---|
| 三组并行（稳健/性能/优雅） | 性能派在严格 TS 类型约束下系统性失败，1/3 成本打水漂。改为两组。 |
| 多模型同源辩论（五巨头内部辩论） | 同源模型多角色是伪多样性，ROI 为负 |
| 运行时代码热加载 | 生成代码是静态文件，不在生成阶段执行 |
| 非 TypeScript 语言支持 | 2.0 聚焦 TS + DI 场景 |
| IDE 插件 | 留给 4.0 |
| 多轮自动修复失败生成 | 成本不可控，重试 1 次后报错退出 |
| 替代现有 DI 框架 | 生成注册代码，不自带容器 |
| 测试生成时读取实现代码 | 会导致 AI 把 bug 当 feature 测，测试必须只看接口签名 |
