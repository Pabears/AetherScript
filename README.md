# AetherScript Fleet Architecture

> 一套用多角色 AI 舰队协作生成高质量 TypeScript 代码的工程方法论。
>
> 本文档是最终产出——不依附于任何具体项目，可直接复用到任何 TypeScript 代码生成场景。

---

## 核心思想

**用 sub-agent 角色战队替代单一 AI**，通过真实的观点碰撞、并行实现 PK、外部锚点校验，系统性提升生成代码的质量。

三个关键原则：

1. **Input 大方投，Output 切细切小。** Input token 便宜，Output token 珍贵且有硬上限。把所有背景塞进 input，把 output 切成最小有效单元（每个 sub-agent 只输出一件事）。

2. **抽象粒度是质量的前置门禁。** 一个类的实现超过 150 行，说明这个类承担了太多职责——这是设计问题，不是 prompt 问题。在设计阶段强制拆分，而不是在生成阶段用行数限制强行压缩。

3. **质量闭环需要外部锚点。** AI 审 AI 不是质量保证。唯一可信的校验点是确定性的、不可被幻觉欺骗的：编译通过 + 测试运行通过。

| 角色 | 代号 | 模型 | 职责 |
|------|------|------|------|
| 🤪 超级狂野PM | `pm` | Gemini 3.1 Pro (temp 1.4) | 挖需求、脑洞、用户真实意图 |

**五巨头** = 全部五个角色（需求拷问、最终放行）

## 完整流程

### Phase 1: aesc-pre（需求与架构）

#### Step 1 — PRD 拷问（五巨头并行）

五个角色同时 spawn，各自从自己的视角提出 3-5 个最刁钻的问题：

```
sessions_spawn(
  task="你是[角色]。原始需求：[需求]。提出3-5个最刁钻的问题，给出你认为最大的风险点。输出≤400字。",
  label="aesc-pre-[角色代号]",
  model="[角色对应模型]"
)
```

收集五份输出 → 用户回答 → 5轮 PK 收敛（连续2轮无新观点则提前结束）→ 输出 `docs/PRD.md`

#### Step 2 — Technical Design（三巨头 PK）

基于 PRD，三巨头各自提出接口设计方案，5轮 PK 收敛：

- **输出内容**：模块边界、接口签名（TypeScript 类型）、依赖关系、不可 autogen 的安全组件清单
- **不包含**：Abstract Class 代码、实现代码
- **输出文件**：`docs/TECHNICAL-DESIGN.md`

#### Step 3 — Abstract Classes（含复杂度门禁）

按 Technical Design 逐类生成 Abstract Class，每类一个 sub-agent。

**复杂度门禁（硬性规则，不通过不继续）：**

| 指标 | 阈值 | 动作 |
|------|------|------|
| 方法数 | > 3 | 必须拆分 |
| 全部方法参数总数 | > 8 | 必须拆分 |
| 预估实现行数 | > 150 | 必须拆分 |

拆分不是"行数限制"，是设计问题：一个类做了太多事，必须在设计阶段解决。

---

### Phase 2: aesc-gen（并行实现与裁决）

**每个类独立处理，类与类之间并行。**

#### Step 4 — 两组并行实现

| 团队 | 风格 | 优先级 |
|------|------|--------|
| 🏗️ 稳健派 | 防御性编程 | 完整错误处理、边界检查、可维护性 |
| 🎨 优雅派 | 设计模式 | 可读性、可扩展性、最小复杂度 |

两组均使用 Opus。

**Prompt 必须包含框架提供的代码骨架，LLM 只填方法体，不允许修改骨架结构。** 这是防止"不继承接口/抽象"这类低级错误的唯一可靠方式。

#### Step 5 — 三巨头裁决（每类一个 sub-agent）

```
sessions_spawn(
  task="你是三巨头合体（arch+qa+sec）。评审以下两个实现，选出最优或指定融合点。输出严格限600字。",
  label="aesc-gen-verdict-[类名]",
  model="claudeopus"
)
```

- Output 严格限 600 字，格式固定（评分表 + 选型 + 融合点 + 安全问题）
- 最高分 < 60 → 融合或重新实现
- 结果写入 `src/generated/[name].impl.ts`

---

### Phase 3: aesc-test（TDD 测试）

**测试必须先于实现生成，且禁止看到实现代码。**

#### Step 6 — 测试生成（每类一个 sub-agent）

```
sessions_spawn(
  task="你是QA黑客。只看以下抽象类签名和JSDoc，生成黑盒测试。禁止猜测实现细节，只测接口契约。输出≤150行。",
  label="aesc-test-[类名]",
  model="claudeopus"
)
```

- Input：只有抽象类签名 + JSDoc，**禁止传入 .impl.ts**（AI 看到实现后会把 bug 当 feature 测）
- Output：测试框架格式，≤150 行

#### Step 7 — 强制运行（唯一外部锚点）

```bash
bun test --bail
```

- ✅ 通过 → 写入 `test/`，继续
- ❌ 失败 → 阻断，报告失败用例，修复 impl 或重新生成

**不运行 = 不存在。** 这是整个流程唯一不可被 AI 幻觉欺骗的质量校验点。

## Sub-agent 运行规范

### 并行 vs 串行

- **并行**：同一步骤内的多个 agent（如五巨头同时拷问、每个类同时裁决）
- **串行**：步骤之间必须串行（Step 1 → Step 2 → Step 3 → ...）

### 生成与写入分离

**sub-agent 只负责生成内容，不负责写文件。** 主 session 收集输出后，自己调用 write 工具写入。

理由：sub-agent 在 output token 即将耗尽时执行 write 调用，content 参数会被截断，文件无法落地。

## 版本演进记录

| 版本 | 关键变化 | 触发原因 |
|------|---------|---------|
| v1.0 | 单 agent 流程 | 初始版本 |
| v2.0 | 五巨头 PRD + 三组并行 + 三巨头裁决 | 提升代码质量 |
| v3.0（当前） | Technical Design 节点、两组并行、TDD、复杂度门禁 | 实战暴露：性能派全灭、截断问题、测试质量为零 |

### v3.0 变更的原因

**砍性能派**：性能派（Gemini 高温度）在严格 TypeScript 类型约束下 7/7 不继承抽象类，1/3 成本打水漂。"多样性"的前提是每组都有赢的可能。

**加 Technical Design 节点**：原流程 PRD 直接跳到 Abstract Class，缺少接口推导层。Technical Design 是让三巨头在写代码之前对齐模块边界的机会。

**复杂度门禁**：截断问题的根本原因不是 prompt 工程，是抽象设计功能太集中。在设计阶段强制拆分，从根上消灭截断。

**TDD + 强制运行**：测试从未真正运行 = 质量保证为零。AI 审 AI 是闭环幻觉，唯一可信的是跑通测试。
